# 自动化成熟度：什么时候可以让 Cron 跑起来

> 很多用户会问："做到什么程度可以让 cron 自动跑？"答案是：不需要做到完美，L2 就够。

## 三个层次

| 层次 | 工具 | Cron 能做什么 | 需要人吗 |
|------|------|---------------|----------|
| L1 基础 | 基础查询 + 统计 | 固定操作 + 简单汇总 | 每次都要人看产出 |
| L2 标准 | L1 + 任务模板 | 按模板执行任务，产出结果 | 偶尔抽查 |
| L3 智能 | L2 + LLM 推理 | 深入分析 + 异常判断 + 主动告警 | 基本不管 |

## 各层次的投入

| 层次 | 封装时间 | 建议频率 | 风险 |
|------|----------|----------|------|
| L1 | 2-4 小时 | 每 30 分钟 | 产出质量不稳定，需要人解读 |
| L2 | 1-2 天 | 每 1-2 小时 | 模板覆盖不全时有遗漏 |
| L3 | 3-5 天 | 每 2-4 小时 | LLM 成本高，可能过度分析 |

## Capricorn 应该怎么引导

### 用户想等"全部做完"再启动

告诉用户：
> 不需要等所有工具都完美。先用现有工具跑一份产出，看看效果，再迭代。跑起来比憋大招强。

### 用户不确定当前到了哪个层次

判断方法：
- 只有基础工具能查数据 → L1
- 有基础工具 + 任务模板，能按固定结构产出 → L2 ✅ 可以启动 cron
- 有完整 skill，能做推理和判断 → L3

### 用户问"什么时候可以基本不管"

满足这两个条件就行：
1. 任务模板跑通，产出格式稳定
2. 质量检查通过率 > 80%

## 推荐路径

```
L1（今天）→ 手动跑 1 个任务，验证能出产出
  ↓
L2（明天）→ 加任务模板，设 cron 每 2 小时
  ↓ 跑 1-2 天，看质量
质量稳定 → 减少人工介入
质量不稳 → 调整模板，verifier cron 纠偏
```

核心原则：**先跑起来，边跑边改。**
