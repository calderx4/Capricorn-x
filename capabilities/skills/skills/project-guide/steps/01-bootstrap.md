# 步骤 1.1：理解需求

> 用户描述要做的事情，Capricorn 澄清并沉淀为 requirements.md。

---

## Capricorn 应该做什么

1. **倾听用户描述** — 用户说"我们要做XX"
2. **提问澄清** — 不清楚的地方主动问：
   - 信息源是什么？在哪？
   - 目标受众是谁？产出给谁看？
   - 有没有已有的标准或规则？
   - 哪些是重点关注维度？
3. **写 requirements.md** — 用用户能看懂的语言写需求文档
4. **给用户确认** — "我理解的需求是这样的，对吗？"

## 产出

```markdown
# requirements.md

## 项目目标
{一句话描述}

## 信息源
- 类型：{数据库/文档/API/文件}
- 来源：{描述}

## 任务要求
- {要求1}
- {要求2}

## 关注维度
- {维度1}
- {维度2}

## 产出对象
- {给谁看}
```

## 完成标志

- [ ] requirements.md 已写入工作文件夹
- [ ] 用户确认理解正确

# 步骤 1.2：试做样本 + 沉淀规则

> 和用户一起做几个样本任务，把反馈沉淀为 alignment.md。

## Capricorn 应该做什么

1. **选一个最典型的任务** — 从用户描述中挑一个最基础的任务
2. **完整执行一次** — 用现有的 tools 跑完
3. **展示结果** — 给用户看原始结果
4. **收集反馈** — 用户说"不行"的地方就是规则
5. **写 alignment.md** — 把反馈转化为对齐规则

## 反馈 → 规则的转化

```
用户说："重点内容要单独拿出来讲"
→ 写入 alignment.md："重点关注项占比 >30% 时，必须单独分析"

用户说："要有前后对比"
→ 写入 alignment.md："时间维度必须有前后对比"

用户说："异常要标出来"
→ 写入 alignment.md："偏离常规的值必须标记为异常"
```

## 产出

```markdown
# alignment.md

## 总规则

### 任务要求
- {规则1}
- {规则2}

### 质量底线
- 每个结论必须有具体依据
- 每个发现必须有明确表述
- 异常情况必须标记
```

## 完成标志

- [ ] 至少完成 2-3 个样本任务
- [ ] alignment.md 已写入工作文件夹
- [ ] 用户确认规则正确

# 步骤 1.3：建初始能力

> 根据需求，创建专用的 tools / skills / workflows。

## Capricorn 应该做什么

1. **盘点已有能力** — 已加载的 tools/skills
2. **识别缺失** — 完成任务还缺什么
3. **创建缺失能力**：
   - **Tool**：某个原子操作没有对应工具（如查询、计算）
   - **Skill**：某类任务需要明确的执行流程
   - **Workflow**：多步操作需要编排
4. **展示给用户** — "我建了这些工具，看看行不行"

## 产出

在 `vertical_hub/{vertical}/` 下（或工作区工具目录下）：
- `tools/` — 新增的 .py 工具文件
- `skills/` — 新增的 SKILL.md
- `workflows/` — 新增的 workflow 模块

## 完成标志

- [ ] 至少 1 个 tool 已创建
- [ ] 至少 1 个 skill 已创建
- [ ] 核心执行流程能跑通

# 步骤 1.4：定质量标准

> 基于样本任务的经验，定义"好的产出应该长什么样"。

## Capricorn 应该做什么

1. **回顾样本结果** — 哪些结果用户满意，哪些不满意
2. **抽象检查维度** — 把反馈提炼为可检查的规则
3. **定义检查方法** — 每个维度怎么检查（正则 / 关键词 / 结构匹配）
4. **更新 alignment.md** — 在"质量底线"部分追加检查维度
5. **给用户确认** — "我定的标准是这样的，对吗？"

## 产出

追加到 `alignment.md` 的"质量底线"部分：

```markdown
### 质量检查维度

| 维度 | 检查方法 | 通过条件 |
|------|----------|----------|
| 有标题结构 | Markdown 标题匹配 | ≥ 1 个标题 |
| 有具体数字 | 正则 `\d+\.?\d*%?` | >= 2 个数字 |
| 内容充分 | 字符数检查 | >= 100 字符 |
```

注：领域可在此基础上增加特有维度。

## 完成标志

- [ ] alignment.md 质量底线已更新检查维度
- [ ] 至少 3 个检查维度
- [ ] 用户确认标准合理

# 步骤 1.5：协商自进化策略

> 和用户商定：系统在没人看的时候，可以自主做到什么程度？

## Capricorn 应该做什么

向用户提出三个关键选择：

**1. 触发阈值** — 连续失败几次开始纠偏？（建议 3 次）

**2. 纠偏权限**（三档）：
- 保守：只允许 bia.md
- 中等（建议）：允许 bia.md + 修改 skill
- 激进：允许 bia.md + skill + 新建 workflow/tool

**3. 人工确认级别**：
- 全自动：自动生效，下次 Phase 3 汇报
- 半自动（建议）：bia.md 自动生效，skill 等下次确认
- 全确认：所有改动等 Phase 3

## 产出

```markdown
# evolution_policy.md

## 触发条件
- 任意质量维度连续 ≥ N 次不通过

## 纠偏权限
- 允许：bia.md 追加规则
- 允许/禁止：修改已有 skill
- 禁止：新建 workflow/tool（pending_approval）

## 人工确认
- bia.md 修改：自动/待确认
- skill 修改：自动/待确认
- 新建能力：必须 Phase 3 审批

## 回滚
- 纠偏后连续 2 次仍不改善 → 自动回滚 + changelog 记录
```

## 完成标志

- [ ] evolution_policy.md 已写入工作文件夹
- [ ] 用户确认纠偏权限和确认级别

# 步骤 1.6：拆分子任务

> 把项目需求拆分为可执行的任务列表。

## Capricorn 应该做什么

1. **分析需求** — requirements.md 中的目标需要哪些子任务
2. **拆分任务** — 每个任务粒度适中（一次执行能完成）
3. **排序** — 考虑依赖关系和优先级
4. **写 tasks.md**
5. **给用户确认** — "我拆的任务列表是这样的，有没有漏的？"

## 产出

```markdown
# tasks.md

## 任务列表

### #1 {任务名称}
- 状态：待执行
- 优先级：高
- 说明：{任务描述}

### #2 {任务名称}
- 状态：待执行
- 优先级：高
- 说明：{任务描述}

### #3 {任务名称}
- 状态：待执行
- 优先级：中
- 说明：{任务描述}
```

## 完成标志

- [ ] tasks.md 已写入工作文件夹
- [ ] 每个任务有明确描述和优先级
- [ ] 用户确认任务列表完整
