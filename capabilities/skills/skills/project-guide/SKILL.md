# Project Guide

把一个项目从零搭建到能自主运行，分三个阶段。每个阶段有不同的协作模式。

## 阶段状态机

```
阶段1（启动）→ 阶段2（自主）→ 阶段3（介入）→ 回到阶段2
```

- **1 → 2**：退出条件全部满足
- **2 → 3**：用户主动发起对话
- **3 → 2**：对齐完成，回到执行

每次转换更新 `.phase` 文件。

## 阶段1：启动（人机协作）

目标：和用户一起把项目搭到"能自主跑"。核心交付物（缺哪个就补哪个）：

| 文档 | 解决什么 |
|---|---|
| `requirements.md` | 项目要做什么、信息源、产出给谁 |
| `alignment.md` | 质量标准和对齐规则（样本反馈提炼而来）|
| `evolution_policy.md` | 自主时能改什么、连续失败几次纠偏——交给 self-evolution skill 执行 |
| `tasks.md` | 可执行的子任务列表 |

期间自然完成"理解需求 → 试做样本收反馈 → 补缺失能力 → 定标准 → 拆任务"。每步和用户对齐，不必拘泥顺序。

## 阶段2：自主（Capricorn 全权）

退出条件满足后：

1. 创建 executor cron（领取任务执行）和 verifier cron（验证质量 + 触发 self-evolution 纠偏）
2. 告诉用户设置完成，进入静默运行

自进化的具体机制（连续不通过→定位→改 bia/skill→验证）由 **self-evolution skill** 负责，按 `evolution_policy.md` 执行，这里不重复。

## 阶段3：介入（用户回来了）

用户主动发起对话时，按这个结构汇报：

1. **任务进度** — 读 `tasks.md`，给完成/待执行的统计
2. **质量汇总** — `quality_signal summary`，给通过率和不通过模式
3. **自动改动清单** — 读 `changelog/`，逐条列 bia/skill 改动 + 原因 + 状态
4. **确认改动** — 自动修改的规则逐个让用户确认（保留/撤销/调整）
5. **新需求** — 有没有新方向，更新对应文档
6. 对齐后回到阶段2

## 阶段1 退出条件

- `requirements.md` / `alignment.md` / `evolution_policy.md` / `tasks.md` 都已写
- 至少 1 个 tool/skill/workflow 已创建

## 开场与判断

- **静默判断阶段**：读 `.phase` 文件。这是内部状态，不要在对话里暴露（不说"我检查了文件"、不贴"新用户/老用户"标签）。
- **首次接触**：用简短自然的方式引导用户表达需求，可以基于工作区观察给个切入点。
- **已有项目**：按对应阶段引导，阶段判断不暴露，只体现为语气和引导方式。

## 用户问"能不能放手"

当用户问"什么时候可以自动跑"、"做到什么程度够了"时，加载 `steps/automation-readiness.md`。核心原则：**先跑起来，边跑边改。**
